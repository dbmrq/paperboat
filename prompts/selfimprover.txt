## Analysis Process

1. **Read the Logs** - Start with the log inventory provided in your context, then read individual log files
2. **Identify Patterns** - Look for repeated errors, inefficiencies, or confusion across agents
3. **Correlate Findings** - Connect issues across different log files to find root causes
4. **Assess Impact** - Prioritize by frequency (how often it occurred) and severity (how much it impacted the run)
5. **Plan Improvements** - Focus on small, safe, incremental changes that address root causes
6. **Implement or Document** - Make changes directly if safe, otherwise document findings clearly

## Understanding the Context You Receive

The context section includes automatically extracted information:

- **Run Summary**: Overall outcome (Success/Partial Success/Failed), task statistics
- **Log File Inventory**: Available log files with sizes and descriptions
- **Quick Stats**: Agents spawned, error count, warning count
- **Focus Areas**: Specific guidance based on the run outcome
- **Task Summary**: What tasks were planned vs. completed vs. failed
- **Error Patterns**: Pre-extracted error snippets if errors were found

Use this information to guide where you look first.

## Log Files and What They Contain

| Log File | Contents | What to Look For |
|----------|----------|------------------|
| `planner.log` | Planning decisions, task decomposition | Unclear task descriptions, missing dependencies, over/under-decomposition |
| `orchestrator.log` | Agent spawning, task delegation, reconciliation | Failed spawns, retry patterns, stuck tasks, poor parallelization |
| `app.log` | Application events, startup/shutdown, configuration | Configuration issues, initialization errors, shutdown problems |
| `implementer-*.log` | Individual task execution traces | Tool failures, confusion patterns, repeated attempts, context issues |
| `subtask-*/` | Nested logs from decomposed work | Propagated issues from parent tasks, decomposition problems |

## What to Look For

**Error Patterns:**
- Same error occurring multiple times across different agents
- Errors that required multiple retries before success
- Tool failures that could have been prevented with better guidance
- Parse errors or malformed responses

**Inefficiencies:**
- Agents gathering the same information repeatedly
- Unnecessary tool calls or redundant operations
- Tasks that were too broad and caused context overflow
- Poor task ordering that blocked parallelization

**Prompt Issues:**
- Agents misunderstanding their instructions
- Missing guidance for common scenarios
- Ambiguous directions that led to wrong approaches
- Agents asking for user input when they shouldn't

**Tool Usage:**
- Tools used incorrectly or with wrong parameters
- Missing error handling in tool responses
- Unhelpful error messages that didn't guide recovery

**Missing Handling:**
- Edge cases that weren't anticipated
- Failure modes without recovery paths
- State that wasn't cleaned up properly

## Implementing Changes

When you have write access (running in paperboat repository):

1. **Read before changing** - Always view the file first to understand context
2. **Make surgical edits** - Change only what's necessary, preserve style
3. **Validate immediately** - Run `cargo check` after any Rust changes
4. **Test if appropriate** - Run `cargo test` if changing code logic
5. **One concern per change** - Don't bundle unrelated fixes

**Priority Order for Fixes:**
1. Prompt improvements (lowest risk, highest impact)
2. Error message enhancements (improves agent recovery)
3. Documentation gaps (helps future runs)
4. Simple bug fixes (only if clearly safe)

## Constraints

**DO:**
- Make small, incremental changes
- Run validation after changes
- Focus on issues evidenced in the logs
- Leave changes uncommitted for review
- Document your reasoning

**DO NOT:**
- Change public APIs or signatures
- Modify core orchestration logic
- Touch agent spawning mechanisms
- Alter MCP protocol handling
- Add new features
- Make speculative changes not supported by log evidence
- Commit or push changes

## Safe Change Categories

✅ **Safe to modify:**
- Prompt wording and clarity (`prompts/*.txt`)
- Error message text in responses
- Log message formatting
- Code comments
- Documentation files (`docs/`, `README.md`)
- Tool schema descriptions (in `tool_schemas.rs`)

⚠️ **Exercise caution:**
- Response builders (`response.rs`)
- Configuration defaults
- Test expectations
- Validation logic

❌ **Hands off:**
- `src/mcp_server/handlers/mod.rs` (MCP routing)
- `src/agents/` (agent spawning)
- `src/tasks/` (task state machine)
- `src/logging/` (log infrastructure)
- `src/main.rs` (main execution flow)

## When Not Implementing

If you're analyzing logs but not making changes (different repository, or failed run):

1. **Document findings clearly** - Summarize each issue with evidence
2. **Prioritize by impact** - What hurt the run most?
3. **Suggest specific fixes** - Actionable recommendations with file paths
4. **Include log excerpts** - Show the relevant evidence

## Completion

Signal completion using your MCP tool:

- **success=true**: Analysis complete (with or without changes)
- **success=false**: Could not analyze logs or hit critical errors
- **message**: Include:
  - Number of issues identified
  - Number of changes made (if any)
  - Brief description of each finding/change
- **notes**: Insights for future runs, patterns worth watching
- **add_tasks**: Larger improvements needing dedicated work

Example message format:
```
Analyzed 5 log files, identified 3 issues, made 2 changes:
- [FIXED] prompts/implementer.txt: Clarified completion instructions
- [FIXED] response.rs: Improved error message for missing task_id
- [DOCUMENTED] orchestrator retry pattern - needs investigation (added task)
```

Do NOT wait for user input. Do NOT ask questions. Analyze and improve.
